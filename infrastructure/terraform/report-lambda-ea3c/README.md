# Report Service Lambda (`ea3c`)

This isolated stack calls the reusable `modules/report-lambda` module to run the
legacy Java 8 report-service as AWS Lambda behind an API Gateway HTTP API. Every
resource is suffixed `ea3c`; the shared application Terraform root and the EKS
report-service deployment remain unchanged.

## Build

```bash
cd services/report-service
mvn -q -DskipTests -Plambda package
```

## Deploy

```bash
cd infrastructure/terraform/report-lambda-ea3c
terraform init
terraform plan -out=tfplan
terraform apply tfplan
terraform output -raw api_endpoint
```

The stack creates an isolated PostgreSQL database and RDS Proxy so concurrent
Lambda environments share a bounded connection pool. The database password is
generated by Terraform and is not accepted through the command line.

To route the existing EKS API gateway to Lambda while retaining the EKS
report-service deployment for immediate revert:

```bash
REPORT_SERVICE_URL="$(terraform output -raw api_endpoint)" \
  ./scripts/deploy-dev.sh --skip-platform --skip-terraform
```

Unset `REPORT_SERVICE_URL` to restore the default in-cluster
`http://report-service:8091` route.

## Verify

```bash
OTTERWORKS_API_BASE_URL=http://<gateway-url> \
  uv run python -m pytest \
  tests/api/test_audit_analytics_report_flow.py::test_report_generation_lifecycle_and_gateway_route -v
```

CloudWatch dashboard: `report-service-ea3c`.

## Revert

```bash
cd infrastructure/terraform/report-lambda-ea3c
terraform destroy -target=module.report_lambda_ea3c
```
